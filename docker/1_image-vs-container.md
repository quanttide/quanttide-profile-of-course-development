# 课时 1：镜像 vs 容器

## 1.1 为什么不能直接"运行镜像"？

刚接触 Docker 的人都会有这个困惑：我做了镜像，为什么还要 `docker run` 一下才能用？为什么不让我直接"运行"镜像，就像双击一个 exe 文件那样？镜像不就是拿来运行的吗？

这个困惑的背后，是一个关键的设计选择——镜像和容器是两回事。

## 1.2 先讲虚拟机是怎么做的

在 Docker 出现之前，最主流的应用打包方式是虚拟机。虚拟机的做法是：把整套东西——包括操作系统、应用、依赖——打包成一个巨大的镜像。然后在另一台机器上"启动"这个镜像，等于模拟出一台完整的电脑，在里面跑一个完整的操作系统。

虚拟机的问题是：太重了。启动一个虚拟机要几分钟，占几个 GB 的磁盘空间。而且虚拟机的"镜像"和"实例"是绑死的——启动镜像就是运行这个虚拟机，没有更细的区分。

## 1.3 再讲 Docker 的设计

Docker 的设计者换了一个思路：能不能利用 Linux 内核自带的隔离机制（Namespace + Cgroups），只隔离应用进程，不虚拟化整个操作系统？

这样，打包的东西就不再是一个完整的虚拟机，而是一个**只读的静态快照**——包含应用代码、依赖库、配置文件，但不包含操作系统内核。这个只读的快照，就是**镜像**。

那容器是什么？容器是镜像的**运行实例**。你每 `docker run` 一次，Docker 就在镜像这个只读快照之上，加一层可写的容器层，分配独立的网络和进程空间，启动你的应用。

## 1.4 回看那个困惑

现在再问"为什么有镜像还要 docker run？"答案就清楚了：

- **镜像** — 定义"要跑什么"（只读模板）
- **`docker run`** — 创建"怎么跑"（运行时实例）

一个镜像可以被 `docker run` 无数次，每次创建一个独立的容器，互不干扰。就像一张 CD 母盘可以压制出无数张光盘，每张光盘在不同的播放器里放。

## 1.5 那容器停止后数据去哪了？

容器层是可写的，但容器一旦被删除（`docker rm`），这一层也随之销毁。所以容器内的数据是临时的。如果需要持久化数据，需要用卷挂载（`-v`）把数据写到宿主机磁盘上，绕开容器的生命周期。

这也是镜像和容器分离的另一个体现——镜像负责环境，卷负责数据，容器负责运行。各司其职。
