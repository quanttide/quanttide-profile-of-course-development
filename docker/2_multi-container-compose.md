# 课时 2：多容器协作

## 2.1 单容器够用吗？

学完课时 1，你已经理解了镜像和容器的关系。但在实际项目中，一个应用很少只有一个服务——通常有数据库、后端 API、前端页面、缓存、消息队列……每个服务都需要自己的运行环境。

如果每个服务都手动 `docker build` → `docker run`，启动一个应用就要开好几个终端窗口，还要记住每个容器的端口映射、网络配置、启动顺序。这比不用 Docker 的时候还麻烦。

## 2.2 先讲手动怎么做

在没有编排工具的情况下，你需要：
- 为每个服务写一个 Dockerfile
- 分别 build 每个镜像
- 分别 run 每个容器，手动指定 `--network` 让它们互通
- 按依赖顺序启动（数据库先启动，后端再启动，前端最后启动）

这套流程做一次还行，每天做几十次就受不了了。而且你没法跟同事说"你把我这 6 个终端窗口的命令挨个跑一遍"。

## 2.3 再讲 Compose 的设计

Docker Compose 解决的就是这个问题。它让你用一个 `docker-compose.yml` 文件描述整个应用由哪些服务组成、每个服务的镜像从哪来、端口怎么映射、依赖什么卷、谁先谁后。

然后一行命令 `docker-compose up` 就把所有事情按顺序做完。

关键在于 Compose 没有发明任何新概念——它只是把多个"镜像→容器"的操作组合成一个声明式文件。镜像还是那个镜像，容器还是那个容器，只是不用你手动挨个操作了。

## 2.4 回看那个困惑

现在再问"为什么多容器这么麻烦"？答案：

- 麻烦的不是 Docker，是**手动编排**
- Compose 不改变镜像和容器的关系，只是把"多个镜像→多个容器"的管理自动化了
- `docker-compose.yml` 本质上是把一个项目的多容器拓扑写成了配置文件，可复用、可分享

## 2.5 镜像、容器、Compose 的分工

- **Dockerfile / 镜像** — 定义每个服务"跑什么"
- **容器** — 每个服务的运行实例
- **docker-compose.yml** — 定义这些服务"怎么组合"
