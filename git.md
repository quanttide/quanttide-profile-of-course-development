# Git 课程蓝图

> 版本：2.0
> 状态：已确定

## 核心教学思路

新手学 Git 最大的困惑，不是记不住命令，而是觉得 Git 很"绕"——

**"我在本地提交了一次，为什么还要再 push 一次？这两个不都是保存吗？有什么区别？"**

这个问题是 Git 教学的钥匙。解释清楚它，分布式和集中式的本质区别就透了，后面的所有概念都是这个核心决策的自然延伸。

## 教学流程

### 1. 先讲 SVN（集中式）

在 Git 出现之前，最主流的版本管理工具是 SVN。SVN 是**集中式**的：有一台中央服务器，所有人的代码都存在上面。你想提交，必须先连上服务器，直接在服务器上操作。离了服务器，你什么都干不了——不能提交，不能看历史，不能对比版本。所有操作都集中在服务器上完成。

### 2. 再讲 Git（分布式）

Linus 写 Git 时面对的场景是：Linux 内核有上千个贡献者，分布在全球各地，他们不可能所有人都连同一台服务器工作。

所以他做了相反的设计：**每个人本地都有一个完整的仓库**。你不联网也能提交代码，本地就有一份完整的历史记录，可以在自己的机器上做任何操作——创建分支、查看历史、对比版本、回退修改。远程有没有服务器，跟你没有关系。

### 3. 回看那个困惑

现在再问"为什么本地 commit 了还要 push？"答案就清楚了：

- `git commit` — 提交到**本地**仓库，对自己负责
- `git push` — 把本地提交**复制**到远程仓库，对团队负责

两件事，两个动作。SVN 把它们混在一起（提交即上服务器），Git 把它们拆开了。这不是设计得"绕"，而是给了你更多的控制权——先在自己的世界里自由操作，准备好了再发布给别人。

### 4. 三区操作

理解了分布式设计意图，三区操作就顺理成章了：

- **工作区** — 你正在修改的地方
- **暂存区** — 为什么需要它？因为你想把不同的改动分批次提交，而不是一股脑全塞进去
- **本地仓库** — 本地提交后的历史记录

命令是操作意图的自然结果：`git add`（打包）→ `git commit`（存盘）→ `git push`（同步）。

### 5. 分支单独处理

分支必须与三区操作**分开讲**，混在一起是最大的教学坑。

理解了分布式，分支就很好懂了：SVN 的分支是贵的（在服务器上拷贝代码），Git 的分支是便宜的（本地指针，创建和切换都不需要联网）。所以 Git 社区鼓励大量用分支。

反过来，你也可以理解为什么我们当前的实践是主干分支——所有人都同一条主干上开发。Git 给了你选择的权利，你愿意的话随时可以切到 feature branch 模式。

### 6. 自动化与流程

主干分支模式高度依赖流程自动化，这也是我们大量造自动化工具的原因。

## 关键教学原则

1. **从核心困惑切入**："为什么本地 commit 了还要 push？"——这个问题是分布式 vs 集中式的最佳入口
2. **先讲对比框架**（集中式 vs 分布式），再讲具体操作
3. **把容易混淆的概念拆开讲**（三区操作和分支必须分离）
4. **用真实场景验证**——让学生在理解设计意图的基础上操作，而非死记命令
