# Git 课程蓝图

> 版本：1.0
> 状态：已确定

## 核心思路

Git 的教学不能从命令开始，而要从**设计起源**开始。学生不理解 Git 为什么被造出来，就无法真正理解它的命令在解决什么问题。

## 教学流程

### 1. 溯源：Git 是怎么来的？

- 发明者 Linus Torvalds 在什么场景下写了 Git？
- 当时面对的核心问题是什么？（Linux 内核开发的版本管理需求）
- 当时的替代方案是什么？（SVN — 中心化版本控制）
- 中心化和分布式的本质区别在哪？

### 2. 核心概念对比

先建立**分布式 vs 中心化**这个核心对比框架，再进入具体操作。

- SVN（中心化）：所有人都连同一个中央服务器，历史记录在服务端
- Git（分布式）：每个人本地都有完整仓库，可以不依赖中央服务器工作

### 3. 三个区的设计

从原理倒推操作：

- **工作区**（Workspace）：你正在修改的地方
- **暂存区**（Index/Staging Area）：为什么需要它？——为了让你可以分批次提交，而不是一次性把所有改动打包
- **本地仓库**（Local Repository）：提交后的历史记录

相应地，三个核心命令就自然推导出来了：

- `git add`：把工作区的改动放进暂存区
- `git commit`：把暂存区的改动打包成一个历史记录
- `git push`：把本地历史记录同步到远端

### 4. 分支单独处理

分支问题**必须与上面的三区操作分开讲**，混在一起讲最容易让新手困惑。

- 分支是什么？——一条独立的时间线
- 主干分支（trunk-based / mainline）——所有人在一条分支上工作，就是我们当前的实践
- 分支最佳规范取决于团队协作模式

### 5. 自动化与流程

主干分支模式高度依赖流程自动化——这也是我们大量造自动化工具的原因。

## 关键教学原则

1. **先讲"为什么"，再讲"是什么"，最后讲"怎么用"**
2. **概念对比先行**（分布式 vs 中心化），具体命令随后
3. **把容易混淆的概念拆开讲**（三区操作和分支必须分离）
4. **用真实场景验证理解**——让学生在自己的工作流中体会设计意图
