场景 3 提交修改 ⏱ 5分钟
发布流程第三步：将版本号和 CHANGELOG 的修改提交到 Git，确保工作区干净并生成规范的提交记录。
📋 检查工作区状态
运行 git status 确认工作区干净，没有未暂存或未跟踪的变更。确保只有版本配置文件和 CHANGELOG.md 被修改。
⏱ 1分钟

为当前提交创建版本标签并推送到远程，确保发布版本的唯一性和可追溯性。标签是发布流程中不可重写的锚点，用于稳定标记发布版本。

检查当前分支最近的提交（git log --oneline -1），确认提交内容与预期发布版本一致。根据语义化版本规范或项目约定确定标签版本号，如 v1.2.3 或 scope/v1.2.3。参考 CHANGELOG 或团队讨论确保版本号准

⚠️ 异常 Tag scope 前缀缺失修复 ⏱ 15分钟
当 Tag 属于某个 scope（如 cli）但漏写了 scope 前缀时，需要删除错误的根级别 tag，在正确的位置重建 tag（指向同一 commit），并处理可能存在的孤立 Release。
⚠️ 识别错误的根级别 Tag
查看当前仓库中根级别 Tag 列表，定位缺少 scope 前缀的 Tag（例如误将 'v1.0.0' 而非 'cli/v1.0.0'）。确认该 Tag 对应的 commit 哈希。
⏱ 2分钟
⚠️ 删除错误的根级别 Tag
使用 git tag -d <tag-name> 删除错误的根级别 Tag。例如：git tag -d v1.0.0。若已远程推送，还需使用 git push origin :refs/tags/v1.0.0 删除远程 Tag。
⏱ 3分钟
⚠️ 在正确 scope 下重建 Tag
使用相同 commit 创建带 scope 前缀的正确 Tag。例如：git tag cli/v1.0.0 <commit-hash>。然后推送至远程：git push origin cli/v1.0.0。
⏱ 4分钟

检查是否存在与错误 Tag 关联的 GitHub/GitLab Release。若有，则删除该孤立 Release，并在正确 Tag 下补建 Release（填写相同的版本说明及附件）。
⏱ 4分钟
⚠️ 验证修复结果
本地执行 git tag -l 'cli/*' 确认存在正确 Tag；远程查看 Release 列表，确认孤立 Release 已移除且正确 Release 已创建。最终可执行 git log --oneline -1 cli/v1.0.0
