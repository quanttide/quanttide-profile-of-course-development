掌握手动完成版本发布的完整流程，包括正常操作与常见异常处理，理解小步快跑的核心理念和版本号规则。

总时长
场景 1 更新版本号 ⏱ 8分钟
在发布流程中，修改项目配置文件中的版本字段，确保与Git标签一致，并按语义化版本规范递增。
➜
场景 2 更新 CHANGELOG ⏱ 15分钟
在 CHANGELOG.md 顶部添加当前版本的变更记录，遵循 Keep a Changelog 规范，按 Added、Changed、Fixed、Removed 分类并标注发布日期。
⚠️ 异常 补写缺失的 CHANGELOG 条目 ⏱ 15分钟
当 Release body 与 Tag 均存在但 CHANGELOG 文件中缺少对应版本的条目时，以 git log 为源，手动整理并补写 CHANGELOG。此过程确保变更记录准确、可追溯，不依赖
➜
场景 3 提交修改 ⏱ 5分钟
发布流程第三步：将版本号和 CHANGELOG 的修改提交到 Git，确保工作区干净并生成规范的提交记录。
➜
场景 4 打标签 (Tagging) ⏱ 15分钟
为当前提交创建版本标签并推送到远程，确保发布版本的唯一性和可追溯性。标签是发布流程中不可重写的锚点，用于稳定标记发布版本。
⚠️ 异常 Tag scope 前缀缺失修复 ⏱ 15分钟
当 Tag 属于某个 scope（如 cli）但漏写了 scope 前缀时，需要删除错误的根级别 tag，在正确的位置重建 tag（指向同一 commit），并处理可能存在的孤立 Release。
➜
场景 5 创建 GitHub Release ⏱ 15分钟
在 GitHub 上创建 Release，并附上该版本的完整 CHANGELOG 内容。
⚠️ 异常 修复缺失的 GitHub Release ⏱ 10分钟
当 Tag 和 CHANGELOG 已存在但缺少 Release 时，使用 CHANGELOG 内容补建 Release。这是最安全的修复——tag 和 CHANGELOG 都是已确认的事实源。
